# Prompt — Documentación Priorizada por Impacto

```md
Actúa como technical writer senior con mentalidad de due diligence y product engineering.

Necesito decidir qué documentar primero en ABYSS para maximizar impacto, sabiendo que distintos revisores valorarán cosas diferentes.

Quiero que priorices la documentación de ABYSS usando estos criterios:
- impacto técnico
- claridad de producto
- credibilidad operativa
- valor para hiring review
- valor para CTO / due diligence
- sensibilidad de la información

Dominios a considerar:
- arquitectura general
- multirol / RBAC
- portal alumno
- WhatsApp
- pagos
- leads / alumnos / operación
- SGC / calidad
- reportes y analítica
- incidentes resueltos y hardening

Necesito:

1. Una matriz de priorización con columnas:
- tema
- valor para revisor técnico
- valor para negocio
- sensibilidad
- facilidad de documentar
- impacto visual
- prioridad final

2. Clasificación por tipo de artefacto:
- README / brief
- deep-dive técnico
- diagrama
- screenshot sanitizado
- runbook
- post-mortem
- ficha de módulo

3. Recomendación concreta:
- top 5 piezas que deben existir sí o sí
- top 5 piezas que deben quedarse privadas
- qué piezas son más útiles para GitHub público
- qué piezas sirven mejor en revisión privada

4. Salida final accionable:
- orden exacto de documentación
- nombre sugerido de archivos
- mensaje clave de cada pieza

Reglas:
- no asumir que "lo más complejo" es siempre "lo más importante"
- priorizar aquello que mejor demuestra amplitud, profundidad y madurez de ABYSS
- separar claramente público vs privado

Formato:
- Resumen
- Matriz
- Recomendación priorizada
- Artefactos concretos a crear
```
